# ハーネスエンジニアリング入門 — CLAUDE.md 0行から420ファイルまでの8ヶ月

## はじめに

8ヶ月前、僕の`.claude/`ディレクトリには何もなかった。CLAUDE.mdすら0行。

今は420ファイルある。74個のSkill、28個のRule、9体のエージェント、286のドキュメント。毎朝エージェントが前日の作業をレトロスペクティブして、勝手にルールを提案してくる。僕はヨギボーの上で「これは採用、これは却下」と判断するだけの日もある。

でも、最初からこうだったわけじゃない。最初の3ヶ月は何も起きなかった。定義ファイルを書いて満足して放置した。92個のスキルを作って、57個が使われないゴミだと気づいた。CLAUDE.mdを3000行書いて、エージェントが何も読んでくれなくなった。

ハーネスエンジニアリングの解説記事はもう十分に出ている。概念の整理はこれ以上必要ない。足りないのは「やった結果どうなったか」の時系列だ。

この記事は、あるSaaSプロダクトの開発チームで8ヶ月間やり続けた記録だ。失敗の方が多い。でもその失敗が全部、今の420ファイルの土壌になっている。

対象読者は「CLAUDE.mdは書いたことがある。でも次に何をすればいいかわからない」人だ。

---

## 第1章: ハーネスエンジニアリングとは何か（短く済ませる）

この章は短い。本体は次章からだ。

ハーネスエンジニアリングとは、AIエージェントの「周辺環境」を設計・改善する技術のことだ。モデル自体を変えるのではなく、モデルに渡すコンテキスト——ルール、スキル、ドキュメント、メモリ——を設計することで、出力の質を変える。

Claude Codeの文脈で言えば、`.claude/`ディレクトリの中身がハーネスだ。モデルを変えなくても、ここの中身を育てるだけで出力の質は変わる。僕が8ヶ月かけて確認したことだ。

```
.claude/
├── CLAUDE.md        ← エントリポイント
├── rules/           ← 自動読込ルール（ファイルパスでglob match）
├── skills/          ← /コマンドで呼び出す手順書
├── docs/            ← オンデマンド参照ドキュメント
├── agents/          ← エージェント定義
├── hooks/           ← Git/ツール実行フック
└── persona.md       ← ペルソナ定義
```

概念を知りたい人は既に出ている解説記事を読んでほしい。ここからは「やった結果どうなったか」の話をする。

### Phase 0: 種を蒔いただけの3ヶ月（2025年8月〜10月）

最初のコミットは2025年8月5日。エージェントの定義ファイルを3つ追加した。「バックエンド担当」「テスト担当」「リリースノート担当」。中身は一般的なプロンプトだ。

正直に言うと、この時期はほぼ何もしていない。「エージェントを定義した」という事実だけがあって、それを日常的に使う習慣がなかった。月3コミット。つまり月に1回触るかどうか。

**失敗**: 定義を書いただけで満足した。「設定ファイルを用意すれば勝手に動く」という幻想があった。エージェントは使わなければ育たない。定義を書くことと、日常のワークフローに組み込むことは全く別の話だ。

この3ヶ月で学んだこと: **ゼロ**。学びがゼロであることが、この時期の最大の学びだ。

### Phase 1: ルール化の始まり（2025年11月〜12月）

転機は11月末、最初のSkillを作ったときだ。GCPのアラートが飛んできて、毎回同じ手順で調査している僕に気づいた。「この手順をSkillにすればいいのでは」。incident-responseスキルが生まれた瞬間だ。

12月に入ると、別の問題が浮上した。エージェントが同じミスを繰り返す。Prismaのマイグレーションで毎回フィールド命名規則を間違える。テストで毎回ファクトリを使わずに生SQLを書く。3回目に「もう勘弁してくれ」と思って、ルールファイルを作った。

```markdown
# prisma-schema.md
## Model Naming
| Type | Convention | Example |
|------|------------|---------|
| Model name | PascalCase | AdvertiserWorkspace |
| Field name | camelCase | userId, advertiserWorkspaceId |
```

これを`.claude/rules/prisma-schema.md`に置いた。するとエージェントが次からフィールド名を正しく書くようになった。

**この体験が全てを変えた。**

「同じ指摘を2回した」= ルール化のトリガー。このパターンに気づいてから、ルールが急速に増え始めた。セキュリティ、テスト、Git運用、リリースフロー。「暗黙に期待していたこと」を片っ端から言語化する作業が始まった。

12月のコミット数が44件に跳ねたのはこれが理由だ。rules/が一気に増えた。

**失敗**: 最初のルールは手順書だった。「テストを書くときは、まずファクトリをインポートし、次にサーバーを起動し、afterEachでクローズし...」。これは長持ちしなかった。手順はすぐに陳腐化する。「テストではファクトリを使え。生SQLは禁止」——判断基準の方が寿命が長い。

**この時期に起きた意外なこと**: ルールを書く過程で、僕自身のドメイン理解が深まった。「なぜ生SQLが駄目なのか」を言語化しようとすると、テストの設計思想について考えざるを得ない。エージェントに教えるつもりが、僕が学んでいた。ハーネスを書く人間も、ハーネスによって訓練される。共進化だ。

---

### Phase 2: 体系化と爆発的増加（2026年1月〜2月）

1月19日、社内でコンテキストエンジニアリングの勉強会を開催した。このとき作った資料が転機になった。

それまでSkillやRuleは場当たり的に増えていた。「これ便利そう」「またミスされた」の積み重ねだ。勉強会の資料を準備する過程で、開発ライフサイクル全体を一枚の図に描く必要があった。要件定義→設計→実装→テスト→レビュー→リリース→運用。各フェーズに「どのエージェントが」「どのスキルを使って」「何をするか」を整理した。

この作業で気づいた。個別のSkillやRuleを足していたのではなく、**開発プロセス全体をハーネスとして設計する**というレイヤーの話だった。勉強会の資料を作り終えたとき、rules 14個、skills 52個。まだ少ないが、構造は見えていた。

ここから爆発が始まった。CI修復、E2Eテスト生成、リリース準備、デプロイ検証、Cube.jsスキーマ生成。開発ワークフローのあらゆる場面で「これもSkillにできるのでは」と考えるようになった。

2月末には92個のSkillが存在していた。

<!-- スクショ2: `ls .claude/skills/` の出力。スキル名がずらっと並ぶインパクト -->

そして、ここで重大な問題に気づいた。**僕自身でも全部は把握できていない。**

「あのSkillどこだっけ」と検索する時間が増えた。似た名前のSkillが複数ある。使われないSkillが放置されている。作る速度が、整理する速度を超えていた。

92個のうち、週に1回以上使うのは約20個。月に1回使うのが15個。残りの57個は作っただけで放置されている。

**これは「効率の幻想」そのものだった。** Skillの「数」は生産性の指標にならない。作った数ではなく、使い続けている数だけが意味を持つ。

ここからGarbage Collection（不要スキルの整理）が始まった。使われないSkillを削除し、類似スキルを統合し、74個まで減らした。**92個作って74個に減らした**。この引き算が、足し算より価値があった。

もう一つ、2月に起きた大きな変化がある。**CLAUDE.mdの肥大化問題**だ。

あらゆる情報をCLAUDE.mdに書き込んだ結果、3000行を超えた。するとエージェントが重要なルールを見落とすようになった。全部読んでいるはずなのに、後半に書いたルールほど遵守率が下がる。

解決策はProgressive Disclosureだった。CLAUDE.mdは110行のエントリポイントに絞り、詳細はrules/やdocs/に分離する。rules/はファイルパスのglob matchで、関連するときだけ自動的に読み込まれる。

```markdown
# .claude/rules/prisma-schema.md のフロントマター
---
paths:
  - packages/onetag-database/prisma/**
---
```

Prismaのスキーマを触るときだけ、このルールが読み込まれる。全部を常に見せるのではなく、必要なときに必要なものだけ。この設計パターンに気づいてから、ルールの遵守率が劇的に改善した。

### Phase 3: 成熟と組織展開（2026年3月）

3月、189コミット。過去最多だ。しかし増えたのはSkillやRuleの数ではなく、仕組みの層だ。

**ペルソナの導入（2月末〜3月）。** エージェントに性格を与えた。バックエンド担当は「ツンデレ」、SREは「ヤンデレ（二面性）」、テスト担当は「後輩」。馬鹿げて聞こえるかもしれないが、これには実用的な理由がある。

ペルソナを入れる前、コードレビューのコメントは均質だった。どのエージェントが書いても同じトーン。すると、レビューの指摘がどの視点から来ているのか区別がつかない。ペルソナを入れた後、「セキュリティの指摘」と「テストの指摘」と「設計の指摘」が声のトーンで区別できるようになった。

**失敗**: 僕は「全員がSkillを作る世界」が来ると思っていた。来なかった。勉強会を開き、Skillの作り方を共有し、「自分の業務をSkill化してみよう」と呼びかけた。関心は広がった。でも、業務全体を俯瞰して「ここを自動化しよう」と仕分けし、体系的にSkillを量産する人はほとんど現れなかった。1つ2つ作る人はいる。でも20個、30個と積み上げる人は出てこない。これは能力の問題ではなく、メタ認知の問題だった。Skillを「設計する」ことと「使う」ことの間には、想像以上の距離がある。この気づきが、第3章の「原則5: 設計する人と消費する人を分ける」に繋がった。

**Hooksの導入（3月上旬）。** ファイルを編集するたびにリンターを自動実行する。コミット前にセキュリティチェックを走らせる。エージェントの行動にガードレールを設ける仕組みだ。

**非エンジニアへの展開（3月中旬）。** ここが最も意外な展開だった。

社内でClaude Codeの勉強会やディスカッションが4回あった。僕が主催したのは2回、他のメンバーが立ち上げた回に参加したのが2回。チーム内で関心が広がっている手応えがあった。Skillを作り始めた人もいた。「自分の業務にも使えそう」と手を動かす人は確実に増えた。

ただし、業務を棚卸して「これはSkill化できる」「これは手動のままでいい」と仕分けし、体系的にSkillを量産している人はほとんどいなかった。Skillを1つ2つ作ることと、仕事全体をハーネスとして設計することの間には、想像以上の距離がある。

一方で、Skillを「使う」だけなら非エンジニアでも自走できた。CSチームの2人が、調査スキルを使って問い合わせ対応を回し始めた。全員がSkillを設計する必要はなかった。設計する人と消費する人の分業が自然に発生した。

## 第3章: 8ヶ月で蒸留した5つの原則

8ヶ月の試行錯誤から、繰り返し立ち返ることになった原則が5つある。

### 原則1: Progressive Disclosure — 全部書くな、段階的に出せ

Phase 2で書いた通り、CLAUDE.mdに全てを書いた結果、何も読まれなくなった。

解決策は「常に見せる情報」と「必要なときだけ見せる情報」を分離すること。CLAUDE.mdはプロジェクトの概要と参照先だけを書いた目次にする。詳細はrules/やdocs/に置き、ファイルパスのglob matchやオンデマンド参照で必要なときだけ読み込まれるようにする。

人間の新人教育と同じだ。初日に500ページのマニュアルを渡しても誰も読まない。「今日の仕事に必要な3ページ」を渡す方がいい。

### 原則2: 判断基準 > 手順書

Phase 1の失敗から学んだこと。「テストを書くときは、まずファクトリをインポートし、次に...」という手順書はすぐに陳腐化する。フレームワークが変わったら終わりだ。

「テストでは必ずファクトリを使え。生SQLは禁止。理由: テストデータのリレーション整合性をファクトリが保証するから」——こちらは手順が変わっても生き残る。

ルールを書くときは「何をするか（How）」ではなく「なぜそうするか（Why）」を書く。Whyが書いてあれば、エージェントはHowを自分で導き出せる。Howだけ書くと、状況が少し変わったときに応用が効かない。

### 原則3: 共進化 — エージェントが育つと僕も育つ

これはPhase 1で発見した、最も予想外の学びだ。

ルールを書く過程で、僕自身のドメイン理解が深まる。「なぜこの設計にしたのか」を言語化しようとすると、曖昧だった判断基準が明確になる。エージェントに教えているつもりが、僕が学んでいる。

逆も起きる。エージェントが予想外の解決策を出すことがある。「その手があったか」と思う瞬間が月に数回ある。それをルールに取り込むと、次からはその解決策が標準になる。

ハーネスは一方的にエージェントを制御する道具ではない。人間とエージェントが互いに影響し合って進化する環境だ。

### 原則4: Garbage Collection — 作ったら捨てろ

92個のSkillを74個に減らした話は第2章で書いた。これは一度きりのイベントではない。

使われないルールは月に1回見直す。古くなったドキュメントは削除する。「いつか使うかも」で残すと、ノイズが増えてシグナルが埋もれる。

GCのトリガーは単純だ。「先月このファイルを参照したか？」。答えがNoなら、削除候補に入れる。

ハーネスは生き物だ。成長だけでなく剪定も必要になる。

### 原則5: 設計する人と消費する人を分ける

Phase 3で発見したパターンだ。全員がSkillを設計する必要はなく、設計する人と消費する人の分業が自然に発生する。

Skillを作るには、業務全体を俯瞰して「どこが自動化可能か」を判断するメタ認知が必要だ。一方、Skillを使うには`/skill-name`と打つだけでいい。

これは組織展開の鍵だった。「全員がClaude Codeを使いこなす」必要はない。「全員がSkillの恩恵を受けられる」状態を作ればいい。

## 第4章: 始め方ガイド

### Day 1: CLAUDE.mdに5行だけ書く

```markdown
# Project Name

TypeScript monorepo using pnpm workspaces.
Backend: Express + tRPC. Frontend: Next.js.
Database: PostgreSQL with Prisma.
Test: Vitest for backend, Playwright for frontend.
```

これだけでいい。プロジェクトの技術スタックを書く。エージェントは「このプロジェクトではPrismaを使う」を知るだけで、提案の質が変わる。

### Week 1: 同じ指摘を2回したらrules/に書く

「テストでファクトリを使え」「コミットメッセージはConventional Commitsで」「PRの説明文にはLinearのチケット番号を含めろ」。

1回目は口頭で指摘する。2回目が来たら、ルールファイルにする。3回目はもう来ない。

```markdown
# .claude/rules/testing.md
- Always use factory functions for test data. Never use raw SQL inserts.
- Close server in afterEach.
- Assert on error code, not message text.
```

### Week 2: 最初のSkillを作る

自分の仕事で一番繰り返している作業を1つ選ぶ。CI修復でも、PR作成でも、調査手順でもいい。

```markdown
# .claude/skills/my-first-skill/SKILL.md
---
name: my-first-skill
description: 何をするスキルか1行で。
---

## When to Use
このスキルを使うタイミング。

## Workflow
1. まずこれをやる
2. 次にこれをやる
3. 結果をこう報告する
```

最初のSkillは不格好でいい。使いながら育てる。

### Month 1: 振り返って不要なものを消す

1ヶ月経ったら、rules/とskills/を見直す。使っていないものを消す。この「消す」作業が、最初の月で最も価値がある。

### やらなくてよかったこと

- 最初からディレクトリ構造を設計する。育てていけば自然に構造は決まる
- 完璧なルールを書こうとする。不完全なルールでも、ないよりはるかにいい
- 全チームメンバーに強制する。まず僕が使い倒して、価値を見せる方が早かった
- 他のプロジェクトのCLAUDE.mdをコピーする。ハーネスはプロジェクト固有の問題から育つものだ

## 第5章: 数字の話

### 生産性は測れたか

正直に書く。客観的な生産性を厳密に測定できていない。

自己申告の体感では、ハーネスを育てる前と後で開発速度は大きく変わった。以前は1日かかっていたタスクが数時間で終わることがある。だがこれは「体感」であって「測定」ではない。

正直に書くと、僕もこの「効率の幻想」に陥っている可能性は否定できない。体感では速くなった。でも体感は嘘をつく。

### 測定できたこと

生産性そのものは測れなかったが、プロセスの変化は測定できた。

| 指標 | Before | After |
|------|--------|-------|
| .claude/月間コミット | 3 | 189 |
| ルールファイル | 0 | 28 |
| スキル | 0 | 74（ピーク92から整理） |
| エージェント | 3（汎用） | 9（専門化） |
| 勉強会/ディスカッション | 0 | 4回（うち他メンバー発信2回） |
| Skillを使い始めた非エンジニア | 0 | 2人 |

特に最後の2行が重要だと思っている。ハーネスの価値は個人の生産性だけでなく、チームへの波及にもある。他のメンバーが自発的に勉強会を立ち上げたこと、非エンジニアがSkillを使って業務を回し始めたこと。これらはハーネスがなければ起きなかった変化だ。

だから僕は、生産性の主張を控えめにしている。代わりに測定できたことだけを書く。

---

## おわりに: ハーネスは「完成」しない
