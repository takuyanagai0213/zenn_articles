# 「Claude Code for X」の空白地帯 ── なぜ日本では誰もやらないのか

## YC W26の風景

2026年冬、Y Combinatorのバッチに採択された198社のうち、168社がAIファーストだった。56社がAIエージェント。ソロファウンダーが27人。

この数字を眺めていて、ひとつのパターンに気づく。

勝っているのは「汎用AI」ではない。**「Claude Code for X」**——つまり、特定の業界ワークフローにAIを深く埋め込んだプロダクトだ。法務のための、不動産のための、医療のための。汎用を捨てて、狭く深く潜った人たちが、投資家の目に留まっている。

YCの分析から見えた2つの勝ちパターンがある。

| パターン | 特徴 | 追従難易度 |
|---------|------|-----------|
| **Pattern A**: 制約を解く | 技術的なボトルネックを突破する（高速化、新アルゴリズム） | 中（技術が追いつけば再現可能） |
| **Pattern B**: ドメイン知識で勝つ | 業界ワークフローを深く理解し、AIで置き換える | **高（蓄積は1日では作れない）** |

Pattern Bが圧倒的に追いかけにくい。そしてこれが「Claude Code for X」の本質だ。

## 日本では誰がやっているのか

では、日本ではどうだろう。

AIツールの解説記事は山ほどある。プロンプトの紹介、便利なプラグインの比較、ChatGPTの使い方入門。でも、**自分の業務ドメインにAIエージェントを100本以上組み込んで、日常業務を回している人は？**

探してみた。

ハーネスエンジニアリング——AIエージェントを業務プロセスに統合する設計手法——の実践者を探すと、出てくるのは解説記事だ。「こういう考え方がある」「こう使えるかもしれない」。概念の紹介で止まっている。

- 100を超えるスキルファイルで日々の開発・運用・リリースを回している事例 → **見つからない**
- 一人ASP事業をAIエージェントで自動化している事例 → **ない**
- コンサルティング業務をAIで構造化して、クライアントごとのナレッジを蓄積している事例 → **ない**

## 3つの壁が同時に立っている

理由は「AIが使えないから」ではない。ツールはもう十分に揃っている。

壁は3つあって、それが**同時に**立っている。

### 壁1: 技術の壁

Claude CodeやMCPを使いこなすには、エンジニアリングの素養がいる。プロンプトを書くだけでは足りない。

具体的に何が必要かというと:

```
.claude/
├── skills/          # 102個のスキルファイル（業務手順の型）
│   ├── release-prep/
│   ├── alert-response/
│   ├── pr-morning-report/
│   └── ...
├── rules/           # 26個のルールファイル（制約と判断基準）
│   ├── security.md
│   ├── testing.md
│   └── ...
├── agents/          # 10体のサブエージェント定義
│   ├── backend-developer.md
│   ├── sre-engineer.md
│   └── ...
└── docs/            # ドメイン知識のコンテキスト
    ├── design/      # 96本の設計書
    ├── decision/    # 75本のADR
    └── ...
```

スキルを設計し、ルールを整備し、エージェント間の連携を組み立てる。これは「AIを使う」のではなく「AIが動く環境を設計する」仕事だ。

### 壁2: ドメインの壁

業界ワークフローを知らなければ、何を自動化すべきかわからない。

例えば、アフィリエイトASPの運用には以下のような業務がある:

- トラッキングリンクの生成とパラメータ管理
- コンバージョンの検知・承認・否認ワークフロー
- 広告主・メディア間の成果レポート
- 不正検知とフィルタリング
- 請求・支払いサイクルの管理

これらは教科書に書いていない。3年、5年と現場にいて初めて見える景色がある。「このCSVのこのカラムが空の場合、過去にこういう障害があったから特別な処理が必要」といった暗黙知は、ドキュメントにも書かれていない。

### 壁3: 関係性の壁

クライアント固有のデータ、業界の暗黙知、信頼関係に基づくフィードバック。これらはAPIでは取得できない。人間が時間をかけて築いたものだ。

この3つのうち、2つを持っている人はそこそこいる。

| 組み合わせ | 人物像 | 足りないもの |
|-----------|--------|------------|
| 技術 + ドメイン | 業界特化のエンジニア | クライアントとの関係性 |
| ドメイン + 関係性 | 業界歴の長い営業・コンサル | エージェント設計の技術 |
| 技術 + 関係性 | フリーランスのエンジニア | 深いドメイン知識 |

3つ全部を持っている人は、極端に少ない。そして「Claude Code for X」は、3つ全部を要求する。

## コピーできるものと、できないもの

面白いのは、この3つの壁のうち、設計図に相当する部分はコピーできるということだ。

スキルファイルの構造、エージェントの設計パターン、MCPサーバーの接続方法。これらはGitHubに公開すれば誰でも読める。実際、僕のスキル設計は[Zennの記事](https://zenn.dev/takuyanagai0213/articles/claude-code-100-skills-full-record)で全て公開している。

でも、**コピーできないもの**がある。

- 3年間の運用で溜まったポストモーテム20件
- 障害対応の判断パターン
- 33,999件のobservation（AIが蓄積した文脈記憶）
- 月175本のPRを通して蓄積された「この変更はここに影響する」という感覚

設計図は1日でコピーできる。蓄積は1日では作れない。

だからこそ、今この瞬間に始めている人と、来年始める人の間には、1年分のギャップではなく、**1年分の蓄積のギャップ**が生まれる。この差は時間が経つほど広がる。

## 実際に何が起きるか

102個のスキルで業務を回すと、こういうことが起きる:

```
朝: /ohayo → カレンダー・Slack・Linear・Gmailを統合した朝ブリーフィング
午前: /pr-morning-report → 3リポジトリの全PRをスキャン、担当者別に整理してSlack投稿
午後: /alert-response → Sentryエラーを自動調査、根本原因を特定してLinearチケット起票
夕方: /release-prep → リリースアイテム収集、SEV分類、チェックリスト生成
夜: /oyasumi → 1日の作業サマリーと翌日プレビュー
```

これは「便利ツール」の話ではない。**業務フロー自体がエージェントの上で動いている**状態だ。

1つ1つのスキルは単純なMarkdownファイルだ。でも102個が連携して、組織の業務知識とMCPサーバー経由の外部サービス（Slack, Linear, GitHub, BigQuery, GCP）に接続されると、一人でチームの仕事を回せるようになる。

## 余白

僕がやっているのは、ある業務ドメインで100を超えるスキルを組み上げて、PR全体の68%を一人で回すような仕組みだ。派手な話ではない。毎日の地味な改善の積み重ねにすぎない。

でもこの領域には、ほとんど誰もいない。

技術的に不可能なわけではない。ツールは揃っている。ドメイン知識を持つエンジニアも日本にはたくさんいる。

たぶん、足りないのは「やってみる」という一歩だ。解説記事を書く側から、自分の業務を丸ごとAIエージェントに載せる側へ。その一歩を踏む人が、まだ極端に少ない。

この空白地帯に、あなたは何を見るだろうか。
