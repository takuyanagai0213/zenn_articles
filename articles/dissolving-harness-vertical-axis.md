# 融解するハーネス — watanyさんの『混乱と私見』への、1年実運用からの補足

watanyさんが書かれた[「AIエージェントの"ハーネス"に関わる混乱と私見」](https://zenn.dev/watany/articles/d8b692bbca65a3)を読んだ。

論点整理の鋭さ、そして「話半分に読んでください」という謙虚な閉じ方に、深く共感した。

この記事は、watanyさんの記事への**補足**だ。反論ではない。ベンダー側の人間でもない。アフィリエイト広告システムのプロダクト開発(設計・開発・運用)に3年間携わりながら、個人として Claude Code に 100のSkills、33,999件のエージェントメモリ、420ファイルのハーネスを育ててきた実装者の、現場からの追記だと思って読んでほしい。

わたしもこの1年間、この「混乱」のど真ん中で実装していた。名前がつく前から、黙々と育てていた。

## watanyさんの論点(わたしの理解)

記事を読んで、論点は3つに整理できると理解した。

**論点1: 「ハーネス」という言葉が二重の意味で使われている**

- **内部ハーネス**(作り手視点): LLM以外の実装部分。LangChainの「Agent = Model + Harness」、Anthropicの「LLMの長時間駆動馬具」
- **外部ハーネス**(使い手視点): エージェントへのフィードバック環境。Mitchell HashimotoやOpenAIの語り口

同じ「ハーネス」という言葉が、違うスコープで使われている。

**論点2: ベンダーの製品戦略への懸念**

各ベンダーが自社製品の立場からハーネスを定義している。watanyさんは特にAnthropicに対して、「記事の通りに読者が参考にしても妨害されるんじゃねえの?」という疑念を表明している。Cline対Claude Code、Coworkへの定額適用禁止などの具体例を挙げて。

**論点3: ロックイン志向 vs 現場実務志向**

ベンダーの言葉を鵜呑みにすると、ロックインに巻き込まれる。実務者は自分で判断する必要がある。

---

以上が、わたしのwatanyさんの論点の理解だ。ここに1年実運用者として3つ補足させてほしい。

## 補足A: 「融解するハーネス」には、もう一つの軸がある

watanyさんの「内部/外部ハーネス」は、**モデルからの距離 × 作り手/使い手**という軸での整理だ。分析として極めて有効だ。

ただし、1年間手を動かしてきた実装者の視点で見ると、**もう一つの軸での融解**が見えてきた。

**垂直軸: 思想 ↔ ハーネス ↔ インフラ**

この縦軸で、境界は日々融解している。

### 思想が凝固してハーネスになる

わたしがClaude Codeに100のSkillを育てる過程はこうだ。

1. ある日、同じ判断を3回繰り返していることに気づく(これは**思想/美学**)
2. 判断パターンを言語化して .md ファイルに書き出す(思想が形式化の入口に来る)
3. Skillとして登録し、名前をつける(**ハーネス**に凝固する)
4. 使ううちにSkillの粗さが見えてくる(ハーネスが思想を更新する)

つまり、**上位の思想が下位のハーネスを形作り、ハーネスの動きが思想を逆流更新する**。この融解の循環が、1年かけて100個のSkillに結晶化した。

### ハーネスがインフラに融解する

同様に、ハーネスとインフラの境界も動いている。

2026年3月、AnthropicがManaged Agents(MA)を発表した。わたしが自前のハーネスで組んでいた「vault、Session永続化、Brain/Hands分離」が、**MAというインフラに吸収された**。

これまで自分のハーネスだった部分が、ベンダーのインフラに移る。**上の層(ハーネス)が下に押し下げられ、わたしはその上の層(思想・ドメイン知識)に集中できる**ようになった。

この「押し下げ」は、watanyさんが懸念する「ベンダーの製品戦略」とは別の文脈だ。わたしの実装を脅かすのではなく、わたしが専念すべき層を明確にする。境界は動きながら、上に集中する余裕を作る。

### 33,999件のエージェントメモリが、融解の証拠

memory/ ディレクトリに33,999件のエージェントメモリが蓄積されている。これは「思想の記録」でもあり「ハーネスの動き」でもあり「インフラへの観察」でもある。

あるエージェントメモリを見ると、それがどの層のものか曖昧だ。

- 「リリース直前に不安が強くなる自分に気づいた」→ 思想
- 「デプロイ手順のSkillを作った」→ ハーネス
- 「MAのSession永続化が使える」→ インフラ

でもこれらは全部、**同じ融解の循環の中にある**。境界で切ることに意味がない。

### 二つの軸を重ねて初めて、ハーネスの全体像が見える

watanyさんの軸(作り手/使い手)と、わたしの軸(思想/ハーネス/インフラ)を重ねると、ハーネスが多層構造に見えてくる。

watanyさんの懸念(ベンダー戦略)は、主に「作り手側の思想とハーネス」の交差点に集中している。ここはベンダーが売りたい場所だ。

一方、実装者としてのわたしの戦場は「使い手側の思想」の階層だ。ここは誰にも売れない。個人の1年間の蓄積でしか作れない。

watanyさんの二分法は**分析のための横軸**、わたしが育てた1年は**融解の縦軸**。両方を重ねると、ハーネスという概念の広さが見えてくる。

## 補足B: 個人実装者のベンダー非依存性

watanyさんのベンダー戦略への懸念は、正しい。

「Anthropicの記事を読んで実装したら、結果的にAnthropicの製品に依存させられる」という警戒は、ビジネス視点として完全に妥当だ。

ただ、**個人実装者としての1年間の経験から言うと、ハーネスは意外にもベンダー非依存で構築できる**ことが見えてきた。

### `.claude/` 配下は、思想が本体、実装は移植可能

わたしのハーネスは `.claude/` ディレクトリに格納されている。中身の主要構成は:

- 100のSkillsは、ほとんどが `.md` ファイル(プロンプト構造とツール呼び出しのレシピ)
- 28のRulesは、`.md` ファイル(条件+指示)
- 9体のAgentsも `.md` ファイル(役割定義)
- memory/ は SQLite + Markdown
- settings.json はJSON

これらは**Claude Code固有のディレクトリ構造に依存しているが、中身の思想は他のAIモデルでも再現可能**だ。実際、Skillの大部分は単なるMarkdown形式の手順書なので、CursorでもAiderでもClineでも、プロンプトとして食わせれば同じ動作をする。

33,999件のエージェントメモリはSQLiteベースで、どの環境にも移植できる。

### ロックインの本質は「思想」ではなく「データとフロー」

ベンダーロックインが本当に怖くなるのは、**思想を奪われた時ではなく、データとフローを握られた時**だ。

Claude Codeに依存しているように見えて、わたしの実装は以下を全て自分で持っている:

- Skill の定義(Markdown)
- Rule の定義(Markdown)
- Agent の定義(Markdown)
- Memory(SQLite + Markdown)
- ワークフローの連携(Bash、Python、Makefile)
- ハーネスの蓄積(1年分のコミット履歴)

Claude Codeが明日突然終了しても、このデータとフローは自分のGitHubリポジトリにある。別のツールに移植するコストは、ゼロではないが、ロックインと呼ぶほどではない。

だから、watanyさんのロックイン懸念に対して、わたしはこう補足したい。

**「ベンダーの言葉を信じるのではなく、自分の実装物を信じろ」**

実装物は裏切らない。ベンダーは裏切る可能性がある。だから、ベンダーがどういう戦略を取ろうと、自分の実装物が残る設計にしておく。これが1年運用して得た結論だ。

## 補足C: 実装は、用語論争の先に走っている

watanyさんの記事は、「用語の混乱」を整理する試みだった。これは業界への大きな貢献だ。

ただ、現場の実感として追記させてほしい。

**用語が混乱している間にも、実装は先に走っている。**

### 名前がつく前からやっていた

わたしがClaude Codeで100のSkillsを育て始めたのは、2025年末。「ハーネスエンジニアリング」という言葉がまだ広まっていない頃だった。

- 100 Skills(2026年3月時点、Zenn記事で公開済み)
- 33,999件のエージェントメモリ(3.5ヶ月で蓄積)
- 420+ファイルのハーネス構造(8ヶ月で構築)
- 3年間のアフィリエイト広告システム運用経験(業務フローをAI化する実践)

名前がないから躊躇する、という理由はなかった。「この作業を自動化したい」「この判断をパターン化したい」という痛みの駆動だけで、実装は進んだ。

### 混乱している暇があれば、実装しよう

watanyさんが混乱を整理してくれたことで、業界の解像度は上がった。でもわたしは、**整理を待ってから実装を始める必要はないと思っている**。

混乱している時期こそ、実装の自由度が最大だ。名前がついた瞬間、ベンダーが「私たちのハーネスが標準です」と言いに来る。それまでに自分の実装物を積み上げておけば、どんな標準化の波が来ても、自分の蓄積は残る。

watanyさんの「話半分に読んでください」という謙虚さは、まさにこの姿勢と同じだ。言葉を信じすぎず、実装で応える。

## 閉じ: 言葉は変わる、融解の循環は残る

watanyさんが提示した「内部/外部」の二分法は、**分析ツールとして有効**だ。作り手と使い手の関心領域を横軸で整理したことで、業界の解像度は確実に上がった。

一方で、1年実運用者のわたしから見ると、この横軸とは別に**縦軸の融解**がある。思想が凝固してハーネスになり、ハーネスがインフラに押し下げられ、エージェントメモリが層をまたいで流れ続ける。境界は日々動いている。

もしかすると、ハーネスという言葉は、**名前がついた瞬間に終わる概念**かもしれない。

ベンダーの製品戦略に取り込まれ、標準化され、フレームワーク化され、最後はコモディティ化する。数年後には、ハーネスエンジニアリングという言葉は古くさく響くかもしれない。

でも、その時にも、**個人の蓄積は残る**。

わたしの100のSkillsは、呼び方がハーネスだろうとエージェントワークフローだろうと、Skillsそのものとして動き続ける。33,999件のエージェントメモリは、呼び方が変わっても、個人のコンテキストmoatとして機能する。

**融解の循環だけが、言葉を超えて残る**。思想が凝固してハーネスになり、ハーネスが動いて思想を更新する。この循環が続く限り、個人の蓄積は増え続ける。

watanyさんの「話半分に読んでください」のリスペクトを返しながら、わたしはこう結びたい。

**「言葉は変わる。実装は残る。融解の循環を楽しみながら、手を動かし続けよう」**
