## TL;DR

- Claude Code auto-fix（クラウド版）が発表され、CI自動修正・レビュー自動対応が公式機能になった
- AIによるコードレビュー評価を3層に分けて整理した。プロバイダーが吸収する層と、自分たちにしか作れない層がある
- **層3（ドメインeval）だけがmoat（堀）になる**。ここにコンテキストを注ぎ込むのがコンテキストエンジニアリング

## 何が起きたか

2026年3月27日、AnthropicのNoah Zweben氏がXに投稿した。

> Claude Code auto-fix -- in the cloud. Web/Mobile sessions can now automatically follow PRs - fixing CI failures and addressing comments.

https://x.com/noahzweben/status/2037219115002405076

PRのCI failureを自動修正し、レビューコメントにも自動対応する。開発者は離席していてOK。61万表示、4,800いいねを記録した。

同じ日、僕のチームのスプリントレビューで、AIによるPRレビュー評価ツールが紹介された。その議論から生まれた整理がこの記事の内容だ。

## eval 3層モデル

AIがPRをレビュー・評価する仕組みを3つの層に分けて整理した。

```
┌─────────────────────────────────────────────┐
│ 層1: 実装品質eval                            │ ← プロバイダーが吸収
│  バグ検出、テスト通過、Lint、型チェック       │
│  → auto-fix, CodeRabbit, Cursor等           │
├─────────────────────────────────────────────┤
│ 層2: アーキテクチャeval                      │ ← 均一化が進む
│  設計パターン準拠、ベストプラクティス         │
│  → ベストプラクティスDBの拡充で汎用化        │
├─────────────────────────────────────────────┤
│ 層3: ドメインeval                            │ ← ここだけがmoat
│  自社仕様書との整合性、業界規制への準拠       │
│  過去のインシデントから学んだ安全基準         │
│  顧客固有の要件、暗黙のビジネスルール         │
└─────────────────────────────────────────────┘
```

### 層1: 実装品質eval → プロバイダーが吸収する

コードにバグがないか、テストが通るか、Lintエラーがないか。これはauto-fixやCodeRabbitがやっていること。

Claude Code auto-fixの発表は、まさにこの層がプロバイダーの標準機能になったことを意味する。ここを自前で作り込んでいたチームの努力が、アナウンス1つで陳腐化した。

### 層2: アーキテクチャeval → 均一化が進む

設計パターンに従っているか、ベストプラクティスに沿っているか。これも各社が蓄積しているベストプラクティスDBが充実するにつれて均一化される。

例えば「Server Componentsでデータフェッチすべき」「tRPCのprotectedProcedureを使うべき」といった判断は、フレームワークの公式ドキュメントに書いてある。AIはこれを読める。

### 層3: ドメインeval → ここだけがmoat

**あなたの会社の仕様書との整合性**。これはプロバイダーが知り得ない情報だ。

具体例を挙げる。僕のプロジェクトでは `.claude/` ディレクトリに以下の構造でドメイン知識を蓄積している。

```
.claude/
├── rules/          # 27ファイル — コードレビュー時の判断基準
│   ├── security.md      # RLS必須、tenantProcedure + role check等
│   ├── prisma-schema.md # Migration reversibility matrix
│   └── release.md       # 7-axis release review checklist
├── docs/
│   ├── domain/     # ビジネス用語辞書（「媒体情報格納」「バックデート承認」等）
│   ├── design/     # 96件の機能仕様書
│   └── decision/   # 75件のADR（なぜその判断をしたか）
├── skills/         # 92個の自動化スキル
├── agents/         # 9つのサブエージェント定義
└── reports/
    └── postmortems/ # 20件のインシデント分析
```

これがドメインevalの「燃料」になる。例えば:

- `rules/security.md` に「新テーブルにtenant列があればRLSポリシー必須」と書いてある。CIでは検出できない。設計判断だから
- `reports/postmortems/` に「Prisma Client deployed before migration → tracker downtime」と書いてある。同じパターンのPRが来たら即座に検出できる
- `docs/domain/glossary/` に業界固有の用語定義がある。仕様書の曖昧な表現を正しく解釈するために必要

**層3のevalは、このコンテキストがないと機能しない。** そしてこのコンテキストは、その会社で働いた人間の判断の蓄積でしか作れない。

## プロバイダーが構造的に攻められない理由

AnthropicもOpenAIも「全ユーザーの80%に価値がある機能」を作る。層1と層2はまさにそれだ。

では層3はどうか。

| プロバイダーが出せないもの | 理由 |
|:---|:---|
| あなたの会社のN=1の業務知識 | 1社のためだけに機能を作らない |
| 過去のインシデントから学んだ安全基準 | あなたの会社のポストモーテムは公開されていない |
| 顧客との合意事項・暗黙のビジネスルール | 契約書やSlackのやり取りの中にしかない |
| チーム固有の意思決定パターン | 「うちはLTVを最優先する」という思想 |

プロバイダーは水平展開（全ユーザー共通）しか攻められない。垂直展開（特定企業固有）は構造的に不可能だ。

```
プロバイダー = エンジン（水平・汎用・スケール）
攻めてこない = 燃料と運転（垂直・固有・蓄積）
```

## コンテキストエンジニアリングという生存戦略

ドメイン知識を構造化し、AIに食わせて増幅する技術。僕はこれを**コンテキストエンジニアリング**と呼んでいる。

実践としてやっていることは単純だ:

1. **判断のたびに「なぜそう判断したか」を書く** → ADR（`docs/decision/`）
2. **インシデントのたびに「何が起きて何を学んだか」を書く** → ポストモーテム（`reports/postmortems/`）
3. **業界固有の用語を定義する** → 用語集（`docs/domain/glossary/`）
4. **レビューの判断基準を明文化する** → ルール（`rules/`）

3ヶ月で386ファイル、30,000のオブザベーション記録。これが層3のevalを機能させる燃料になっている。

## 「やれるけどやらない」という判断

auto-fixの発表を見て、改めて思う。

CI自動修正、レビューコメント自動対応。僕もやりたかった。GitHub上にも同様のプロジェクトがいくつも公開されていた。でも手を出さなかった。

**「背筋がいいもの（汎用的に正しいもの）は、公式が出す。だからそこに注力するのは不毛」**

「デモ映えする」「すごいと言われる」機能ほど、プロバイダーが本気で取り組む領域だ。そこで勝とうとするのは、層1の中で差別化しようとする行為であり、構造的に報われない。

注力すべきは層3。プロバイダーが構造的に攻められない場所にコンテキストを積み上げること。

## まとめ

```
層1（実装品質）: プロバイダーに任せる
層2（アーキテクチャ）: 均一化を受け入れる
層3（ドメイン）: ここに全力を注ぐ ← これがmoat
```

コンテキストエンジニアリングは地味な作業だ。ADRを書く、ポストモーテムを書く、用語を定義する、ルールを明文化する。どれも「すごい」とは言われない。

でもこの蓄積だけが、LLM時代に残る堀になる。
