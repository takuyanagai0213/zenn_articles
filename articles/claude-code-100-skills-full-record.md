## 変曲点の物語

### 2024年8月: 月2 PR

手でコードを書いていた。PRを出すのは週1回あるかないか。Claude Code（当時はまだ初期版）を使い始めたが、「ちょっとした補完ツール」程度の認識だった。

### 2025年11月: 最初のSkill

2025年11月27日、最初のSkillを作った。`incident-response`——GCPアラートが来たときの調査手順をスキル化したものだ。

きっかけは単純で、深夜にアラートが来るたびに毎回同じ手順（ログ確認→メトリクス確認→影響範囲特定→Slack報告）を手で打っていたのが面倒だったから。

```
# 最初のスキルコミット
2025-11-27 feat(.claude): add incident-response skill for GCP alert investigation (#3438)
```

この1個が全てを変えた。「反復作業をスキル化する」というパターンを覚えたことで、次々とスキルが生まれ始めた。

### 2025年12月: スキルがスキルを生む

月62 PR。ここで**自己強化ループ**が回り始めた。

```
retro（振り返り）
  → 新しいパターンを発見
  → doc-ship（ドキュメント反映）
  → 次のセッションでエージェントが賢くなる
  → さらに効率的に作業
  → retroで新しい発見
  → ...
```

特に`retro` → `doc-ship`のループが強力だった。毎セッション終わりに`/retro`を実行すると、「このセッションで学んだこと」をClaude Code自身が分析して、`.claude/`への更新を提案してくれる。それを`/doc-ship`でPR化してマージする。翌日のセッションでは、昨日学んだことをエージェントが最初から知っている状態で始められる。

### 2026年2月: 日平均6.8 PR

月214 PR（monopoly-source単体）。3リポジトリ合計で月175 PR。

この頃になると、朝の始業から退勤まで、ほぼ全ての作業にスキルが関わっていた。


Boris Cherny（Googleのエンジニア、Claude Code活用で有名）のチーム20人が日平均5 PR。1人で6.8 PRを出していた。

.claude/に420ファイル。組織内の2位のリポジトリ（61ファイル）の6.9倍。

ただし、これは「頑張って100個作った」のではない。日々の業務で「これダルい」と思った瞬間にスキル化する習慣が身についた結果、気づいたら100個になっていただけだ。

## 失敗したスキル・死んだスキル

100個作って全部生きているわけではない。正直に書く。

### 1. 抽象度が高すぎて使われなかったスキル

初期に作った「汎用的なコードレビュースキル」は、結局使わなくなった。「どんなPRにも使えるレビュー」は結局「どのPRにも刺さらないレビュー」になる。

後に`sweep`（品質・効率・テストカバレッジに特化）と`evaluate`（SEV分類に特化）に分化して、やっと使い物になった。

**教訓**: 汎用的に作るな。具体的なペインに刺せ。

### 2. ツールの進化で不要になったスキル

CI失敗時の自動修正をやるスキルを自前で作っていたが、2026年3月にAnthropicがauto-fixを公式リリースした。自前スキルはお役御免になった。

これは失敗ではなく、むしろ予見された結末だ。プロバイダーが標準機能として出すものを自前で作る必要はない。

**教訓**: プロバイダーが吸収する層（実装品質チェック、Lint修正等）を自前で頑張るのは不毛。自分たちにしか作れないドメインスキルに集中せよ。

### 3. 依存関係が壊れて放置されたスキル

外部APIに依存するスキルで、APIの仕様変更についていけず壊れたものがいくつかある。直すコストより作り直すコストの方が安いので、放置してから新版を作った（`worktree` → `worktree-v2`など）。

**教訓**: 外部依存が多いスキルは壊れやすい。壊れたら直すより作り直す方が速い。

### 4. 「便利そう」で作って使わなかったスキル

「これあったら便利そう」と思って作ったが、実際のワークフローに組み込まれず死んだスキルが数個ある。ペインが先にあるのではなく、ソリューションが先にあるパターンだ。

**教訓**: 「あったら便利」は作らない。「ないと困る」だけ作る。

### 5. 使われるけど改善されないスキル

最初に雑に作って、それ以来一度も改善されていないスキルもある。動いているから放置しているが、出力品質は微妙。ここにGC（ガベージコレクション）の仕組みが必要だと感じている。

## スキル設計の5つの原則

100個作って見えてきたパターンを5つにまとめる。

### 原則1: 業務のペインから始める

技術的に面白いからではなく、「このタスク、毎回ダルい」から始める。

```
良い出発点: 「リリースのたびにSlackに同じフォーマットで投稿するの面倒」
悪い出発点: 「LLMで自動コードレビューできたらかっこいい」
```

良い出発点は`release-notifications`スキルになった。悪い出発点から始めた汎用レビュースキルは前述の通り死んだ。

### 原則2: 最初は雑に作って育てる

最初のバージョンは30分で作る。完璧を目指さない。

`retro`スキルの初期バージョンは「今日やったことを箇条書きにして」くらいの雑なプロンプトだった。使ううちに「.claude/への更新提案も出して」「差分プレビューも見せて」と育っていった。

最初から完璧に設計しようとすると、使う前に飽きる。

### 原則3: retroで毎日学習する

`/retro` → `/doc-ship` のループが最も重要な仕組みだ。

```
セッション終了時:
  /retro
    → 「今日のセッションで、XXXの手順が3回出てきました。スキル化を提案します」
    → 「rules/security.mdに載っていないパターンを発見しました。追記を提案します」
  /doc-ship
    → 提案された変更をPR化
    → マージ
```

人間が意識的に振り返らなくても、エージェント自身が「今日学んだこと」を提案してくれる。これにより、**使えば使うほど賢くなる**サイクルが回る。

### 原則4: 既存スキルを組み合わせる

新しいスキルを一から作るより、既存スキルを組み合わせる方が速い。

```
/release-prep（リリース準備）の中身:
  1. /release-items で変更一覧を取得
  2. /sweep で各PRの品質チェック
  3. /release-notifications でSlackメッセージ生成
  4. Slackに投稿
  5. /release-watch で監視開始
```

`release-prep`は5つの既存スキルの組み合わせだ。新しく書いたロジックはほぼない。

### 原則5: 使わないスキルは捨てる

GC（ガベージコレクション）のスクリプトを用意している。60日間更新のないスキルは「stale」としてフラグが立つ。

```bash
.claude/scripts/gc-audit.sh
# → 60日以上更新のないスキル、ルール、ドキュメントを検出
```

「もったいない」で残すと、エージェントのコンテキストを無駄に消費する。古いスキルは古い前提で動くので、残しておくと間違った結果を出す原因になる。**staleなドキュメントは、ないよりも害がある。**

## これから始める人へ

### 最初の1個は何を作るべきか

「一番ダルい反復作業」をスキル化する。それだけ。

具体的には:

1. 過去1週間を振り返る
2. 「同じ手順を2回以上やったこと」をリストアップする
3. 一番面倒だったものをスキルにする

例えば:
- PRを出すたびに同じフォーマットで説明を書いている → `pr-create`相当
- CIが落ちるたびに同じログを見に行っている → `ci-fix`相当
- 毎朝Slackの未読を確認して今日のタスクを整理している → `ohayo`相当

### CLAUDE.mdに書くべき最低限

プロジェクトのCLAUDE.mdに最低限書くべきことは3つだけ:

### 「100個」ではなく「1個を育てる」

100個は結果に過ぎない。大事なのは、**1個のスキルを育てるサイクルを回す**ことだ。

```
1個作る → 使う → 不便を感じる → 改善する → また使う → ...
```

このサイクルを1個のスキルで回せたら、2個目、3個目は自然に生まれる。「100個作ろう」と目標を立てる必要はない。

## FAQ

### 「全部手書き？」

半分くらいは`/retro`で自動提案されたもの。セッション終了時にClaude Codeが「このパターン、スキル化した方がいいのでは」と提案してくる。人間はそれを判定して、`/doc-ship`でマージするだけ。

もう半分は、業務中に「これダルい」と感じた瞬間に手で書いた。最初の数行を書けば、Claude Codeが残りを補完してくれるので、実質的な手書き量は少ない。

### 「何時間かかる？」

スキル1個あたり30分〜2時間。ただし「育てる時間」は別。

初版は30分で作れる。しかし使い込んで改善を重ねると、1つのスキルに累計5-10時間かかっているものもある（`release-prep`や`retro`など）。

ただし、この投資は確実に回収される。`release-prep`は毎週2回のリリースで使うので、1回30分の節約 × 年100回 = 年50時間の節約。10時間の投資で50時間のリターン。

### 「チームで共有できる？」

git管理なので即共有。`.claude/` ディレクトリごとリポジトリにコミットされている。チームの誰がcloneしても同じスキルが使える。

ただし、問題は「育てる人」の属人性だ。スキルは作った本人が日々使って改善するから良くなる。引き継いだだけでは育たない。ここは未解決の課題。

### 「プロバイダーが標準機能出したら無駄にならない？」

なる。実際にauto-fixが出て、CI修正の自前スキルは不要になった。

しかし、それでいい。プロバイダーが吸収するのは**層1**（実装品質）と**層2**（アーキテクチャ）だ。**層3**（ドメイン知識）——自社の仕様書との整合性、過去のインシデントから学んだ安全基準、業界固有のルール——これはプロバイダーには作れない。

100個のスキルのうち、プロバイダーに吸収されるのはせいぜい10-15個。残りの85個以上は、自社のドメイン知識が詰まった「堀（moat）」になる。
